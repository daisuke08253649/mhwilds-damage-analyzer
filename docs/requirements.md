# 要件定義書
## MHWilds ダメージ解析 Webアプリケーション

**バージョン** 1.0　|　**作成日** 2025年

---

## 1. プロジェクト概要

### 1.1 目的

モンスターハンターワイルズのプレイ動画をアップロードし、AI（OCR）技術を用いてダメージ数値を自動認識・集計するWebアプリケーションを開発する。プレイヤーが自身のプレイパフォーマンスを定量的に分析できる環境を提供することを目的とする。

### 1.2 対象ユーザー

一般公開（デプロイあり）― モンスターハンターワイルズのプレイヤー全般

### 1.3 動作環境

- プラットフォーム：Webブラウザ（PC・スマホ対応）
- デプロイ先：Vercel（フロントエンド）/ Render（バックエンド）

---

## 2. 技術スタック

| レイヤー | 技術・ツール | 備考 |
|---|---|---|
| フロントエンド | Next.js（React） | UIフレームワーク |
| バックエンド | Python / FastAPI | APIサーバー・推論処理 |
| OCR / AIモデル | Gemini API（MVP） → ファインチューニング済みモデルへ移行予定 | 初期はGemini APIで実装・検証。将来的にカスタムモデルへ切り替え |
| モデルホスティング | 移行先は検討中 | MVP後にファインチューニング済みモデルのホスティング先を別途選定予定 |
| フロントデプロイ | Vercel | Next.jsと親和性が高い |
| バックエンドデプロイ | Render | FastAPIサーバーのホスティング |
| 動画処理 | FFmpeg | 動画フレーム抽出 |

## 3. 機能要件

### 3.1 動画アップロード機能

- 対応形式：MP4、MOV、AVI等の主要動画フォーマット
- 最大動画長：50分（目安ファイルサイズは要検証）
- アップロード後、処理ステータスをリアルタイム表示（プログレスバー等）
- YouTubeのURLを入力することで動画を取り込み、アップロードと同様に解析可能

### 3.2 ダメージ認識機能（AIによるOCR処理）

- 動画をフレーム単位に分解し、ダメージ数値テキストをOCRで検出
- MVP段階ではGemini APIを使用してフレーム画像からダメージ数値を認識
- 将来的にはゲームフォント・エフェクトに対応したファインチューニング済みモデルへ切り替え予定（ホスティング先は検討中）
- モデル切り替えを容易にするため、OCR処理部分はインターフェースを抽象化して実装する
- 処理はリアルタイムに近い速度で順次実行（完了まで待たずに逐次結果を返す）

### 3.3 ダメージログ一覧機能

- 時系列でダメージを一覧表示（タイムスタンプ + ダメージ値）
- 何秒時点で何ダメージを与えたかを一覧形式で確認できる
- スクロール可能なログビューアとして実装

### 3.4 結果サマリー機能

- 総ダメージ合計を大きく表示
- 平均ダメージ、最大ダメージ、ダメージヒット数なども表示（任意）
- 結果はCSVまたはJSON形式でエクスポート可能

## 4. 画面構成（UI要件）

| 画面名 | 主な機能・表示内容 |
|---|---|
| ① 動画アップロード画面 | ドラッグ＆ドロップ / ファイル選択でアップロード。処理状況のプログレス表示 |
| ② ダメージログ一覧画面 | 時系列のダメージ履歴を一覧表示（タイムスタンプ・ダメージ値） |
| ③ 結果サマリー画面 | 総ダメージ合計・統計情報の表示。視覚的にわかりやすいレイアウト |
| ④ 解析履歴一覧画面 | ログイン済みユーザーの過去の解析結果を一覧表示。日時・動画名・総ダメージ等を表示し、過去の結果を再閲覧可能 |

**デザイン方針：** シンプル・クリーンなUI。ゲーム系アプリらしくダーク系カラーの採用も検討可。

## 5. 非機能要件・制約

### 5.1 パフォーマンス

- 50分動画（最大）の処理完了目安：15〜30分以内（サーバースペックに依存）
- リアルタイムに近い逐次処理を実装し、全処理完了前から結果を段階的に表示

### 5.2 コスト制約

- 運用コスト：原則0円を目標
- MVP段階はGemini APIの無料枠を活用
- Vercel / Render の無料プランを活用
- ファインチューニング済みモデルへ移行後のホスティングコストは選定後に再検討
- ※ 利用量が増加した場合は有料プランへの移行を検討

### 5.3 認証・ユーザー管理

- 認証の有無に関わらず、アプリの全機能（動画アップロード・OCR解析・結果表示）を利用可能
- ログイン済みの場合：解析結果をDBに保存し、後から見返すことができる
- 未ログインの場合：解析結果は保存されず、セッション終了とともに破棄される
- 認証ライブラリはNextAuth.js（Auth.js）を検討
- サインアップ・ログイン画面はシンプルなメール＋パスワード認証を基本とし、OAuthは将来対応として検討

### 5.4 セキュリティ

- アップロードされた動画ファイルはサーバー上での一時保存に留め、処理後は削除
- APIエンドポイントへの不正アクセス防止（レートリミット等）

## 6. システム構成（概略）

| コンポーネント | 役割 | 技術 |
|---|---|---|
| クライアント | UI表示・動画アップロード・結果表示 | Next.js / Vercel |
| APIサーバー | 動画受信・フレーム抽出・OCR処理オーケストレーション | FastAPI / Render |
| AIモデル | ダメージ数値のOCR認識 | Gemini API（MVP） → ファインチューニング済みモデル（移行先検討中） |
| 動画処理 | 動画→フレーム画像への変換 | FFmpeg（バックエンド内） |

---

## 7. 開発フェーズ（推奨）

| フェーズ | 内容 | 優先度 |
|---|---|---|
| Phase 1 - MVP | 動画アップロード → フレーム抽出 → Gemini API OCR → 結果表示の基本フロー完成 | 最高 |
| Phase 2 - 認証・保存機能 | ユーザー認証（NextAuth.js）・解析結果のDB保存機能の実装 | 高 |
| Phase 3 - UX改善 | リアルタイム進捗表示・ダメージログUI・サマリー画面の充実 | 中 |
| Phase 4 - モデル移行準備 | ファインチューニング用訓練データ収集・アノテーション・学習 | 低 |
| Phase 5 - モデル切り替え | ファインチューニング済みモデルをホスト・Gemini APIから切り替え | 低 |
